# Pulsar — Project Context

> **Бывшее название:** Spicepad (переименовано в Pulsar при добавлении схематического редактора)

## 📋 Описание проекта
GUI-приложение для схемотехнического проектирования и SPICE-моделирования на базе Python + PySide6 + NGspice.
Два режима работы:
1. **Текстовый редактор** — написание и редактирование SPICE-netлистов (.cir, .sp) с подсветкой синтаксиса
2. **Схемотехнический редактор (Lepton EDA)** — визуальное создание схем через внешний `lepton-schematic` с последующим экспортом в SPICE через `lepton-netlist`

## 🏗 Архитектура
**Стек:** Python + PySide6 + NGspice + matplotlib + Lepton EDA (опционально)

### Главное окно (main.py)
- Панель меню: **Файл**, **Вид**
- Панель инструментов (toolbar): Новый проект, Открыть проект, Сохранить проект, Редактор SPICE
- Статус-бар внизу окна — уведомления о конвертации схем
- Док-панель (левая боковая панель): дерево проекта без заголовка и кнопок управления
  - Скрытие/показ через **Вид → Дерево проекта**
  - Две ветки: **Схемы (.sch)** (иконка 🔌) и **Netlist (.cir/.sp)**
  - Название проекта выделено жирным шрифтом
  - Правый клик по папке → добавить файл соответствующего типа
  - Правый клик по .sch файлу → **Конвертировать в .cir/.sp…** / **Удалить…**
  - Правый клик / Delete по файлу → диалог удаления (только из проекта / из проекта и с диска)
  - Двойной клик по .sch → открыть в Lepton EDA
  - Двойной клик по .cir/.sp → открыть в текстовом редакторе
- Авто-экспорт .sch → .cir при закрытии Lepton EDA (thread-safe через Qt-сигнал)
- Мониторинг изменений .sch файла через QFileSystemWatcher
- Уведомления в статус-баре: ✅ успех / ⚠️ ошибка / 🔄 процесс конвертации

### Окно редактора (editor/editor_window.py)
- Основная область: текстовый редактор с подсветкой синтаксиса SPICE
- Нижняя панель: терминал с выводом NGspice (с прокруткой)
- Запуск/остановка через меню **Симуляция**
- Схемотехнический редактор: **Схема → Редактировать в Lepton**

**Функционал (текстовый режим):**
- Открытие/сохранение файлов .cir, .sp
- Создание новых схем (шаблон)
- Сохранить как… с выбором расширения (.cir / .sp / все файлы)
- Запуск/остановка симуляции
- Отображение вывода NGspice в реальном времени
- Автоматическое отображение графиков (стиль осциллографа)
- Выбор цветовой схемы редактора и терминала
- Выбор шрифта редактора
- Контроль несохранённых изменений (`*` в заголовке, диалог при выходе)

**Функционал (схематический режим) 🆕:**
- Двойной клик по .sch файлу в дереве → открыть в Lepton EDA
- Создание/редактирование схем в lepton-schematic (.sch формат gEDA)
- Экспорт .sch → SPICE netlist через `lepton-netlist -g spice`
- **Авто-конвертация при закрытии Lepton** — автоматически .sch → .cir с уведомлением в статус-баре
- **Мониторинг .sch файла** через QFileSystemWatcher — авто-экспорт при сохранении в Lepton
- **Конвертация из контекстного меню** — ПКМ по .sch файлу → диалог выбора расширения (.cir / .sp)
- Автоматическая загрузка полученного .cir в текстовый редактор
- **Обновление открытого .cir** в редакторе при изменении .sch (reload_file)
- Проверка наличия Lepton EDA при запуске
- Подсказки по установке Lepton EDA если не найдена
- Thread-safe callback через Qt-сигнал (`_LeptonCloseSignal`) для корректной работы из фонового потока

> ⚠️ **Известные нюансы Lepton EDA:**
> - `lepton-netlist -g spice` возвращает пустой netlist, если компоненты не соединены проводами или нет GND
> - `lepton-netlist -g spice` **игнорирует** `spice-directive-*.sym` символы (берёт только первую директиву) — Pulsar парсит `.sch` файл напрямую через `extract_spice_directives()`
> - Для корректного SPICE-экспорта нужны символы из `/usr/share/lepton-eda/sym/spice/` или компоненты с атрибутами `refdes` и `value`
> - Lepton-schematic **не может** загружать кастомные .sch файлы с нуля — запуск без файла работает корректно
> - ✅ **Исправлено 08.04.2026:** Конвертация `.sch → .cir` теперь использует флаг `-o output.cir` (как в рабочей команде `lepton-netlist -g spice -o out.cir in.sch`), ранее вывод шёл только в stdout

### ⚠️ Правильная последовательность состояний компонента (как в Lepton EDA)
**Реализовано 12.04.2026 в `schematic_editor_new/components/lepton_component.py`**

**Последовательность состояний:**
1. **Клик по компоненту (select)** → весь компонент жёлтый `#ffcc00` (включая пины и выводы)
2. **Зажал ЛКМ + перемещение (drag)** → исходные цвета: корпус зелёный, выводы коричневые, **пины скрыты**
3. **Отпустил ЛКМ** → снова весь жёлтый (компонент остаётся выделенным)
4. **Клик в любом месте схемы (deselect)** → возврат к исходному состоянию: зелёный корпус, коричневые выводы, красные пины

**Ключевые моменты реализации:**
- `itemChange(ItemSelectedHasChanged)` — обрабатывает выделение/снятие выделения
- `mousePressEvent` — сбрасывает `_is_dragging = False` перед началом клика
- `mouseMoveEvent` — устанавливает `_is_dragging = True` при зажатой ЛКМ
- `mouseReleaseEvent` — сбрасывает `_is_dragging = False` и обновляет цвета
- **Приоритет в `_update_colors()`:** `dragging` > `selected` > `обычное состояние`
- **При deselect** (`selected=False`) — **принудительно сбрасываем** `_is_dragging = False` (компонент может не получить `mouseReleaseEvent` если клик был в пустом месте)
- `boundingRect` отступ `extra = 8` — баланс между хит-тестом и перерисовкой

### ✅ Манхэттенская трассировка проводов (12.04.2026)

**Проблема:** Провода между компонентами должны соединяться строго по ортогональной технологии (только горизонталь + вертикаль). Это критически важно для конвертации `.sch → .cir` потому что netlist требует корректной топологии соединений.

**Решение:** Манхэттенская (ортогональная) трассировка — один pivot между двумя сегментами.

**Алгоритм:**
```
HV режим:  (x1,y1) → (x2,y1) → (x2,y2)  — горизонтально, потом вертикально
           ──────────┐
                     │
                     ↓

VH режим:  (x1,y1) → (x1,y2) → (x2,y2)  — вертикально, потом горизонтально
           │
           ↓
           └──────────→
```

**Функция:** `calculate_manhattan_path(x1, y1, x2, y2, mode, grid)` в `connections/temp_wire.py`
- Привязка к сетке: `round(x / grid) * grid`
- Возвращает 3 точки: `[start, pivot, end]`
- Режим переключается клавишей `R` (HV ↔ VH)

**Механика рисования:**
1. Клавиша `N` → жёлтый провод появляется у курсора и следует за мышкой
2. `R` → переключение режима HV ↔ VH
3. Клик на порте A → пин исчезает, провод привязывается к порту, следует за мышкой
4. Клик на порте B → пин исчезает, провод становится синим → соединение создано
5. Esc или ПКМ → выход из режима

**Фиксация пинов:**
- При подключении `add_connection()` → пин скрывается (`setVisible(False)` + флаг `is_connected`)
- При отключении `remove_connection()` → пин появляется
- При перемещении/повороте компонента — пины НЕ появляются обратно

**Обновление соединений при перемещении/повороте:**
- При перемещении компонента: `itemChange(ItemPositionChange)` → `pin.update_connections()` → `connection.recalculate_path()` → новый манхэттенский путь
- При повороте: `rotate()` → `pin.update_connections()` → `connection.recalculate_path()`
- При отражении: `flip_horizontal()` → `pin.update_connections()` → `connection.recalculate_path()`
- `recalculate_path()` — сбрасывает `pivot_points` и строит новый манхэттенский путь от текущих позиций портов

**Файлы:**
- `schematic_editor_new/connections/temp_wire.py` — `calculate_manhattan_path()`, `TempWireItem` с режимом HV/VH, `toggle_mode()`
- `schematic_editor_new/connections/connection.py` — `ConnectionItem.update_path()` использует манхэттенскую трассировку, `recalculate_path()` для пересчёта
- `schematic_editor_new/tools/wire_tool.py` — `WireTool`: клавиша N, R, клик на порте, создание соединения
- `schematic_editor_new/core/view.py` — обработка клавиш N и R в `keyPressEvent()`
- `schematic_editor_new/components/lepton_component.py` — `LeptonPinItem`: `update_connections()`, `is_connected` флаг; `rotate()` и `flip_horizontal()` вызывают обновление соединений
- `schematic_editor_new/components/port.py` — `PortItem.update_connections()` с вызовом `recalculate_path()`

### 🔌 Классический подход к связям между компонентами (13.04.2026)

**Проблема:** Старый `ConnectionItem` хранил ссылки на `PortItem`/`LeptonPinItem` напрямую и получал их координаты через `scenePos()`. При повороте/перемещении компонента координаты пинов не обновлялись корректно — провод обрывался.

**Решение:** Реализован классический подход из примера на Tkinter (пользователь нашёл рабочий код на Python):
- Компонент хранит **абсолютные координаты пинов** (`LeptonSymbolItem.pin_abs_positions` — список `QPointF`)
- При любом изменении (перемещение, поворот, отражение) → `_update_pin_positions()` пересчитывает все абсолютные позиции через `pin_item.scenePos()`
- `ConnectionItem` хранит ссылки на компоненты + индексы пинов (`start_comp`, `start_pin_idx`, `end_comp`, `end_pin_idx`)
- При `update_path()` → берёт координаты из `component.pin_abs_positions[pin_idx]`
- Для `JunctionItem` (у которого нет `pin_abs_positions`) — fallback на `scenePos()`

**Изменённые файлы:**
- `components/lepton_component.py`:
  - `self.pin_abs_positions = []` в `__init__`
  - `_update_pin_positions()` — пересчёт через `pin_item.scenePos()`
  - `_update_pin_positions_and_connections()` — вызывается через `QTimer.singleShot(0, ...)` при перемещении
  - Вызов `_update_pin_positions()` в `rotate()`, `flip_horizontal()`
  - `_update_colors()` — теперь проверяет `pin.is_connected` перед `setVisible(True)`, подключённые пины ВСЕГДА скрыты
- `connections/connection.py`:
  - `__init__` принимает `start_comp, start_pin_idx, end_comp, end_pin_idx` вместо `start_point, end_point`
  - `_register_pin_connections()` / `_unregister_pin_connections()` — регистрация на пинах
  - `get_start_pos()` / `get_end_pos()` берут координаты из `component.pin_abs_positions[pin_idx]`
  - Fallback на `scenePos()` для `JunctionItem`
  - `remove()` вызывает `_unregister_pin_connections()`
- `tools/wire_tool.py`:
  - `mouse_press()` — обрабатывает клики на портах ТОЛЬКО если `is_wire_mode=True` (режим активен через N или меню)
  - `_create_connection()` — извлекает компоненты + индексы пинов из `LeptonPinItem`
  - `_split_connection_at_junction()` — обновлён для работы с junction (`pin_idx = -1`)
  - `_remove_temp_wire()` — удаляет через `self.scene.removeItem()` (избегает предупреждения)
  - `cancel_drawing()` / `_reset_wire_state()` — полный сброс `is_wire_mode = False`

**Ожидаемое поведение режима "Соединение":**
1. ❌ Клик по порту БЕЗ нажатия N — ничего не происходит (режим не активен)
2. ✅ Нажали N → клик по порту R1 → клик по порту R2 — провод создаётся, пины исчезают
3. ✅ Создание соединения завершается автоматически после второго клика
4. ❌ Клик по порту ПОСЛЕ создания — ничего (режим деактивирован)
5. ✅ N снова — режим активен, можно создать новое соединение
6. ✅ Esc во время рисования — режим завершается полностью, нужно снова N

**Нерешённые проблемы на конец сессии:**
- ⏳ **Провод рисуется но соединение не работает** — пины не исчезают, провод не привязан к пинам
- ⏳ **При повороте компонента провод обрывается** — `_update_pin_positions()` использует `scenePos()` но возможно есть проблема с Qt трансформациями
- ⏳ **Нужно проверить что `_register_pin_connections()` корректно вызывает `pin.add_connection()`**
